# HA Automation Maker 사용법

이 애드온은 두 가지 화면을 제공합니다.

- **루틴** — 한국어 문장으로 규칙을 만들면 애드온이 스스로 상태를 구독하고
  실행하는 **자체 엔진**입니다. Home Assistant의 `automations.yaml`은 쓰지
  않습니다.
- **HA 자동화** — 기존 방식대로 트리거·조건·액션 카드를 조합해 Home Assistant
  표준 자동화(`automations.yaml` 기반)를 만드는 화면입니다.

두 화면은 서로 다른 저장소를 쓰므로 완전히 독립적으로 동작합니다. 어느 쪽을
먼저 써도 상관없습니다.

## 1. 첫 실행

1. 애드온을 설치하고 **시작(Start)** 합니다.
2. 필요하다면 애드온 **설정(구성)** 탭에서 옵션을 조정합니다.
   - **로그 레벨**: `debug`/`info`/`warning`/`error` 중 선택(기본 `info`).
   - **LLM 해석 백엔드**: `끄기`/`API 키`/`구독 CLI` 중 선택(기본 `끄기`). 켜면
     로컬 파서가 확신하지 못하는 문장에 한해 AI 도움을 받습니다. 필요한
     키·토큰(Anthropic API 키 또는 Claude 구독 CLI 토큰)도 이 탭에서
     입력합니다. 자세한 내용은 [6. LLM 해석 백엔드](#6-llm-해석-백엔드)를
     참고하세요.
3. Home Assistant 사이드바에 **자동화 메이커** 패널(로봇 아이콘)이 나타납니다.
   보이지 않으면 애드온 정보 화면에서 "사이드바에 표시"가 켜져 있는지 확인하고
   브라우저를 새로고침하세요.
4. 패널을 열면 **루틴** 화면이 기본으로 표시됩니다.
5. 이 애드온은 Home Assistant Ingress로만 접근할 수 있습니다(외부에서 직접
   접속 불가). 로그인 계정은 **관리자 권한**이어야 정상적으로 열립니다.
6. 필요한 권한은 Home Assistant Core API(`homeassistant_api`) 하나뿐이며,
   Supervisor API는 사용하지 않습니다.

## 2. 화면 구성

상단에 탭 3개가 있습니다.

| 탭 | 내용 |
|---|---|
| **루틴** (기본) | 자연어 문장으로 만드는 규칙 목록과 새 규칙 작성 |
| **HA 자동화** | 기존 트리거·조건·액션 카드 편집기(자동화 목록/위저드/편집기) |
| **설정** | 시간대 경계·사람·모드·별칭 등 루틴 해석에 쓰이는 값 관리 |

## 3. 루틴 — 자연어로 규칙 만들기

### 3.1 문장 입력

화면 맨 위에 큰 입력창이 있습니다. "예: 화장실은 5분 동안 움직임이 없으면
환풍기와 조명을 꺼줘"처럼 하고 싶은 동작을 평소 말투 그대로 씁니다. 입력창
아래에는 바로 써볼 수 있는 예시 문장 칩이 있으며, 탭하면 입력창에 그대로
채워집니다. 문장을 쓴 뒤 **[해석]** 버튼을 누릅니다.

### 3.2 해석 확인 카드

해석 버튼을 누르면 입력창 바로 아래에 **해석 확인 카드**가 나타납니다. 원문
문장이 구간(칩)별로 색깔 있는 하이라이트로 나뉘어 표시됩니다.

- 칩 색상은 역할을 나타냅니다 — 트리거(파랑) · 조건(보라) · 대상/액션(초록) ·
  값(주황).
- 칩마다 해석 상태가 다릅니다.
  - **확정**: 색이 채워져 있고 그대로 저장하면 됩니다.
  - **애매함**(노란 테두리): 탭하면 후보 목록이 드롭다운으로 뜹니다. 각 후보는
    이름과 방·기기 같은 보조 설명을 함께 보여주며, "이 중에 없음…" 항목을
    골라 직접 엔티티를 찾을 수도 있습니다.
  - **해석 못 함**(빨간 점선): 탭하면 엔티티 피커가 열려 직접 대상을 골라야
    합니다.
- 후보나 엔티티를 고르면 그 선택이 고정되고(재해석해도 유지) 카드가 새로
  갱신됩니다.
- 카드 하단에는 해석 결과를 한 줄로 요약한 문구가 있고, 이해하지 못한 부분이
  있으면 "이 부분은 이해하지 못했어요: …" 경고가 함께 표시됩니다.
- 하단 버튼: **[다시 해석]**(문장을 고친 뒤 다시 확인), **[루틴 저장]**. 해석
  못 한 부분이 남아 있으면 저장 버튼이 비활성화됩니다.
- 문장 해석에 AI 도움을 받은 경우 카드에 "AI 도움으로 해석했어요." 문구가
  추가로 표시됩니다.
- 카드 맨 아래에는 **[직접 지정]** 버튼이 있습니다. 파서가 문장을 전혀
  이해하지 못했거나 원하는 대로 해석되지 않을 때 누르면, 문장을 단어 단위로
  쪼갠 편집기가 카드 아래에 펼쳐집니다. 자세한 사용법은
  [3.9 해석이 안 되면: 직접 지정](#39-해석이-안-되면-직접-지정)을 참고하세요.

### 3.3 예시 문장 5개

아래 문장들은 애드온이 실제로 인식하는 표현 방식의 예시입니다. 비슷한 구조로
문장을 쓰면 대부분 정확히 해석됩니다.

| 문장 | 해석 결과 |
|---|---|
| "화장실은 5분 동안 움직임이 없으면 환풍기와 조명을 꺼줘" | 욕실 모션 센서가 5분 동안 계속 꺼짐 상태이면(지속 트리거) → 환풍기와 조명을 끕니다 |
| "나와 와이프가 집에 도착하면 현관 조명을 켜줘" | 내가 먼저 집에 있는 상태에서 와이프가 집에 도착하면(마지막 사람이 트리거) → 현관 조명을 켭니다 |
| "여름에 거실 온도가 28도 이상이면 에어컨을 켜줘" | 계절이 여름이고 거실 온도가 28도 이상이면 → 에어컨을 켭니다 |
| "안방은 30분 동안 움직임이 없으면 슬립 모드로 바꿔" | 안방 모션 센서가 30분 동안 계속 꺼짐 상태이면 → 슬립 모드(씬)를 실행합니다 |
| "거실 조명은 새벽시간에 거실에 움직임이 있으면 10%로 켜줘" | 새벽 시간대이고 거실에 움직임이 감지되면 → 거실 조명을 10% 밝기로 켭니다 |

이 외에도 "~하고 ~하면", "~이상/이하", "다른 곳은 ~없으면", "모든 조명을"처럼
여러 조건을 엮거나 범위를 지정하는 표현도 인식합니다. 문장이 애매하게
해석되면 해석 확인 카드에서 후보를 직접 골라 확정할 수 있습니다.

"슬립모드가 켜지면", "…슬립모드이고…"처럼 모드 이름을 트리거·조건으로 쓰는
표현, "~해주고 …하면 …해줘"처럼 한 문장에 조건-액션 쌍을 여러 개 담는 표현도
인식합니다. 자세한 내용과 예시는 각각 [3.7 상태
모드](#37-상태-모드켜짐꺼짐이-있는-변수)와 [3.8 한 문장에 여러 규칙
담기](#38-한-문장에-여러-규칙-담기)를 참고하세요.

### 3.4 루틴 목록

저장한 규칙은 입력창 아래에 카드 목록으로 쌓입니다.

- 카드 제목은 **원문 문장 그대로**(따로 이름을 붙이지 않은 경우) 표시됩니다.
- 부제에는 트리거 종류 아이콘(⏰ 시간 기반 / 📡 상태 기반 / 🌙 모드 기반),
  방 이름, 대상 엔티티 개수가 표시되고, 한 문장에 규칙이 여러 개 들어 있으면
  "규칙 N개"가 함께 표시됩니다([3.8 한 문장에 여러 규칙
  담기](#38-한-문장에-여러-규칙-담기) 참고).
- 우측에는 활성/비활성 토글 스위치가 있습니다.
- 카드 아래쪽에는 마지막 실행 시각과, 오류가 있으면 배지가 표시됩니다
  ("오류로 꺼짐"은 빨간 배지 — [9. 문제 해결](#9-문제-해결faq) 참고).
- 카드 메뉴에서 **▶ 테스트 실행**(조건을 무시하고 액션만 즉시 실행), **편집**,
  **삭제**(확인 모달 후 삭제)를 고를 수 있습니다.
- 상단 세그먼트로 **방별 / 카테고리별 / 전체** 그룹핑을 전환할 수 있고, 방이
  없는 규칙은 "미배정" 그룹에 모입니다.
- 검색창에 입력하면 문장 전문을 대상으로 부분 검색(초성 검색 포함)합니다.

### 3.5 규칙 편집

목록에서 카드 제목을 탭하면 편집 모드로 들어갑니다. 원래 문장이 입력창에
그대로 채워지고, 저장 당시 확정했던 대상(핀)을 유지한 채 다시 해석해 확인
카드가 표시됩니다. 필요하면 문장을 고치거나 칩을 다시 지정한 뒤 저장하면
기존 규칙이 갱신됩니다.

### 3.6 상단 상태 표시

루틴 화면 상단에는 다음이 항상 표시됩니다.

- **엔진 연결 상태 점**: 초록 = 정상 연결됨, 빨강 = 연결 끊김. 30초마다
  자동으로 갱신됩니다.
- **현재 시간대 라벨**(예: "· 지금: 새벽"): 지금 전역 시간 변수가 어느
  시간대인지 보여줍니다.

### 3.7 상태 모드(켜짐/꺼짐이 있는 변수)

"슬립 모드"처럼 켜짐/꺼짐 상태를 계속 기억하는 변수입니다. 문장에서 이렇게
씁니다.

- **트리거**: "슬립모드가 켜지면…", "슬립모드가 꺼지면…" — 모드가 그 상태로
  바뀌는 순간 규칙이 발화합니다.
- **조건**: "…슬립모드이고…", "…슬립모드일 때…", "…슬립모드가 아니면…" —
  지금 모드가 어떤 상태인지로 조건을 겁니다.
- **액션**: "슬립 모드로 바꿔", "슬립 모드 켜", "슬립 모드 꺼" — 모드 상태를
  바꾸고, 설정에서 지정한 씬(있으면)을 함께 실행합니다.

모드는 **설정 → 모드 관리**에서 관리합니다.

- 이름, 초기값(꺼짐/켜짐), 켜질 때 실행할 씬(또는 스크립트) 엔티티를
  지정합니다.
- 이미 저장된 모드는 **현재 상태** 토글이 함께 표시되어, 화면에서 바로
  수동으로 켜고 끌 수 있습니다. 수동으로 켜고 끄면 그 모드를 트리거로 쓰는
  규칙도 똑같이 발화합니다.
- 문장에서 설정에 없는 새 모드 이름을 쓰고 저장하면, 애드온이 초기값
  꺼짐으로 자동 등록해 줍니다("설정에 없던 모드를 새로 추가했어요" 안내와
  함께).

### 3.8 한 문장에 여러 규칙 담기

"~해주고 …하면 …해줘"처럼 한 문장 안에 (조건 → 액션) 쌍이 여러 개 들어갈 수
있습니다. 예를 들어:

> "새벽에 슬립모드이고 거실 모션이 작동하면 거실조명을 10% 켜주고 3분 동안
> 모션이 없으면 꺼줘"

이 문장은 카드 **하나**로 저장되지만, 안에는 규칙이 **2개** 들어 있습니다.

1. 새벽 시간대이고 슬립 모드가 켜져 있는 상태에서 거실 모션이 감지되면 →
   거실 조명을 10% 밝기로 켭니다.
2. (같은) 거실 모션이 3분 동안 계속 없는 상태이면 → 거실 조명을 끕니다.

두 번째 규칙에서 "꺼줘"라고만 하고 대상을 다시 말하지 않아도, 앞 규칙에서
마지막으로 나온 "거실조명"을 그대로 이어받습니다(컨텍스트 상속). 이런 규칙은
루틴 목록에서 부제에 "규칙 2개"로 표시되고, 편집·삭제·테스트 실행도 카드
하나 단위로 이루어집니다.

### 3.9 해석이 안 되면: 직접 지정

파서가 문장을 전혀 이해하지 못했거나, 자동 해석 결과가 마음에 들지 않을 때는
해석 확인 카드 아래의 **[직접 지정]** 버튼을 눌러 문장을 단어(토큰) 단위로
직접 구성할 수 있습니다.

- 문장이 단어 칩으로 쪼개져 나열됩니다. 칩을 탭하면 역할 지정 시트가 열립니다.
- **역할**을 고르면 역할에 맞는 추가 입력이 나타납니다 — 무시 / 트리거 대상 /
  조건 대상 / 액션 대상 / 상태값 / 수치 조건 / 지속시간 / 시간대 / 모드 /
  요일 / 계절 / 값(밝기·온도) / 동작 / 규칙 나눔. 엔티티가 필요한 역할은
  엔티티 피커가, 숫자가 필요한 역할은 값 입력이 함께 뜹니다.
- 역할 목록에는 "규칙 나눔"도 있어서, 그 단어 자체를 순수하게 구분점으로만
  쓰고 싶을 때 고를 수 있습니다. 다른 역할(예: 액션 대상)을 유지한 채로 그
  지점부터 새 규칙을 시작하고 싶다면, 역할은 그대로 두고 시트 아래의
  **"여기서 규칙 나눔"** 체크박스만 켜면 됩니다.
- 상단 **[초안 채우기]** 버튼을 누르면 파서가 짐작한 역할을 자동으로 채워
  줍니다. 이후 틀린 부분만 고치면 됩니다.
- 아래쪽에 실시간 요약과, 아직 저장할 수 없는 이유(트리거나 액션이 빠졌다
  등)가 표시됩니다. 유효해지면 **[루틴으로 저장]** 버튼이 활성화됩니다.

## 4. 전역 시간 변수

문장에서 "새벽시간에", "여름에", "주말에는" 같은 시간 표현은 애드온이 내부적으로
계산하는 전역 시간 변수를 기준으로 해석됩니다.

- **시간대**(5구간): 새벽 · 아침 · 낮 · 저녁 · 밤. 각 시간대가 시작하는 시각은
  기본값이 있으며 [5. 설정 화면](#5-설정-화면)에서 바꿀 수 있습니다.
- **계절**: 봄(3~5월) · 여름(6~8월) · 가을(9~11월) · 겨울(12~2월).
- **요일 구분**: 평일 · 주말 · 공휴일(한국 공휴일 기준, 공휴일이 평일/주말보다
  우선합니다).

시간대 경계를 넘는 순간이나 정해진 시각("밤 9시가 되면" 등)에는 엔진이 해당
시점에 맞는 규칙을 자동으로 다시 평가합니다.

## 5. 설정 화면

**설정** 탭에서 루틴 해석에 쓰이는 값을 관리합니다. 항목을 고친 뒤 하단
**설정 저장** 버튼을 눌러야 반영됩니다.

- **시간대 경계**: 새벽/아침/낮/저녁/밤 각 시간대가 시작하는 시각을 시:분으로
  지정합니다.
- **사람 매핑**: "나", "와이프"는 기본으로 있고, **+ 사람 추가**로 이름을
  늘릴 수 있습니다. 각 이름에 실제 `person.*` 엔티티를 연결해야 "나와
  와이프가 집에 도착하면" 같은 문장이 해석됩니다.
- **모드 관리**: "슬립 모드"처럼 켜짐/꺼짐 상태를 갖는 변수를 추가·삭제합니다.
  각 모드마다 초기값(꺼짐/켜짐)과, 모드가 켜질 때 실행할 씬(scene) 또는
  스크립트(script) 엔티티를 지정할 수 있습니다. 이미 저장된 모드는 현재
  상태를 보여주는 토글이 함께 나타나며, 여기서 바로 수동으로 켜고 끌 수
  있습니다. 자세한 사용법은 [3.7 상태
  모드](#37-상태-모드켜짐꺼짐이-있는-변수)를 참고하세요.
- **별칭**: 표면형 단어(예: "안방 무드등")를 특정 엔티티에 고정합니다. 이름이
  겹치거나 애드온이 잘못 추측하는 엔티티가 있을 때 사용하세요.
- **LLM 해석 백엔드**: 끄기 / API 키 / 구독 CLI 중 하나를 라디오 버튼으로
  고릅니다. 각 항목 옆에 "준비됨" 또는 "키 없음"/"CLI·토큰 없음" 같은 준비
  상태가 표시됩니다. API 키·구독 토큰 자체는 여기서 입력하지 않고 애드온
  옵션(구성 탭)에서 관리합니다. 자세한 내용은 [6. LLM 해석
  백엔드](#6-llm-해석-백엔드)를 참고하세요.
- **AI 학습**: 로컬 파서·패턴 매칭이 모두 실패한 문장을 AI로 표준 문형으로
  바꿔 애드온이 배우게 하는 기능의 on/off 스위치입니다. **기본은 꺼짐**이며,
  켜려면 위의 LLM 해석 백엔드가 API 키 또는 구독 CLI로 준비돼 있어야 합니다.
  자세한 사용법은 [6.6 AI로 분석해서 배우기](#66-ai로-분석해서-배우기학습)를
  참고하세요.
- **학습된 패턴**: AI 학습으로 지금까지 배운 항목(원문·표준 문형·사용 횟수)이
  목록으로 나타나며, 필요 없는 항목을 삭제할 수 있습니다.

## 6. LLM 해석 백엔드

로컬 한국어 파서만으로 대부분의 문장을 해석할 수 있습니다. 다만 파서가 문장의
뜻을 확신하지 못하거나(확신도가 낮음) 일부 구간을 전혀 해석하지 못했을 때,
그 문장에 한해서만 LLM에게 보조로 다시 해석을 요청할 수 있습니다. 이 보조
기능은 끄기 / API 키 / 구독 CLI 세 가지 중 하나로 켭니다.

**최종적으로 적용되는 값은 [5. 설정 화면](#5-설정-화면)의 "LLM 해석 백엔드"
라디오에서 고르고 저장한 값입니다.** 설정 화면에서 한 번도 저장한 적이
없다면, 그 사이에는 애드온 옵션(구성 탭)의 `llm_backend` 값이 대신
쓰입니다. API 키·구독 토큰 자체는 이 두 화면 중 어느 쪽에서도 아니고 항상
애드온 옵션에서만 관리합니다.

### 6.1 끄기(기본값)

아무 것도 설정하지 않으면 이 상태입니다. AI 호출이 전혀 일어나지 않고, 로컬
파서 결과만 사용하며, 외부로 나가는 요청도 없습니다.

### 6.2 API 키

애드온 옵션(구성 탭)의 **Anthropic API 키** 항목에 키를 입력하고,
`llm_backend`를 `API 키`로 두면 파서가 확신하지 못하는 문장에 한해 Anthropic
API를 직접 호출해 보조로 해석합니다.

- 키는 `settings.json`(설정 화면에서 관리하는 값)에는 저장되지 않고, 애드온
  옵션(비밀번호 필드)으로만 관리됩니다.

### 6.3 구독 CLI

Claude Pro/Max 같은 **구독 계정**이 있다면 API 키 없이도 보조 해석을 쓸 수
있습니다. 애드온 이미지에 Claude Code CLI(`claude` 명령)가 함께 들어 있어,
이 CLI를 구독 계정으로 인증한 상태로 호출합니다.

1. 구독 계정으로 로그인할 수 있는 환경에서 Claude Code CLI를 설치하고,
   `claude setup-token` 명령을 실행해 장기간 쓸 수 있는 토큰을 발급받습니다.
2. 발급받은 토큰을 애드온 옵션(구성 탭)의 **Claude 구독 CLI 토큰**
   (`claude_code_oauth_token`) 항목에 붙여넣습니다.
3. `llm_backend`를 **구독 CLI**로 선택하고 저장합니다.

이 백엔드를 지원하기 위해 애드온 이미지 자체에 Claude Code CLI(네이티브
바이너리)가 포함되어 있어, 이미지 용량이 약 250MB 정도 더 커집니다(백엔드를
쓰지 않고 꺼 두어도 이미지 용량은 동일합니다).

> **참고**: Claude CLI는 일반적으로 `ANTHROPIC_API_KEY`(API 키)가 설정되어
> 있으면 구독 토큰보다 API 키를 우선 사용합니다. 이 애드온은 구독 CLI를
> 호출할 때 API 키를 자동으로 제외하도록 처리하지만, 두 값을 함께 넣어두면
> 어떤 자격 증명이 실제로 쓰였는지 헷갈릴 수 있습니다. 구독 CLI를 쓰려면
> Anthropic API 키 칸은 비워 두는 것을 권장합니다.

### 6.4 공통 동작

- 설정 화면에는 선택한 백엔드가 실제로 쓸 준비가 됐는지("준비됨" / "키 없음"
  / "CLI·토큰 없음")가 표시됩니다.
- AI 보조로 채워진 칩은 해석 확인 카드에서 "애매함" 상태로 표시되어, 저장 전에
  사람이 한 번 더 확인하도록 되어 있습니다.
- 호출이 실패하거나 시간이 오래 걸리면 조용히 로컬 파서 결과만 사용합니다
  (오류가 화면에 노출되지 않고, 키·토큰도 로그에 남지 않습니다).
- AI 도움을 받아 해석된 경우 확인 카드에 "AI 도움으로 해석했어요."(어느
  백엔드였는지도 함께) 캡션이 붙습니다.

### 6.5 하이브리드 패턴 매칭(자동, 무료·오프라인)

애드온에는 자주 쓰이는 표준 문형을 모아 둔 **패턴 라이브러리**가 내장되어
있습니다. 로컬 규칙 파서가 문장을 해석하지 못했을 때, 애드온은 AI를 부르기
전에 먼저 이 라이브러리와 대조해 봅니다.

- 문장을 어휘(기기·방·모드·수치)와 기능어(감지되면·켜/꺼 등)로 분해한 뒤,
  이미 아는 표준 문형의 골격과 같은지 맞춰 봅니다. 골격이 일치하면 그 문형의
  정답 구조에 문장의 값(어느 방, 어느 기기)을 끼워 넣어 규칙을 만듭니다.
- **AI 호출도, 인터넷 연결도 필요 없습니다.** 라이브러리는 애드온 이미지에
  포함되어 있고 전부 기기 안에서 처리됩니다.
- 로컬 파서가 이미 문장을 해석했을 때는 절대 끼어들지 않습니다(기존 해석을
  덮어쓰지 않음). 파서가 실패한 문장에 한해서만, 그리고 문형이 확실히 맞을
  때만 채택합니다(액션의 켜/끄기 방향까지 일치해야 하므로 엉뚱한 매칭이
  걸러집니다). 이렇게 흡수된 규칙은 확인 카드에 출처가 **"패턴 매칭"**으로
  표시됩니다.
- 이 기능은 항상 켜져 있으며 별도 설정이 없습니다.

### 6.6 AI로 분석해서 배우기(학습)

로컬 파서와 패턴 매칭이 모두 실패한 문장은, 원한다면 AI에게 **"이 문장을
라이브러리가 아는 표준 문형으로 바꿔 달라"**고 시킨 뒤 그 결과를 애드온이
**학습**하게 할 수 있습니다. 한 번 배우면 같은/비슷한 문장은 다음부터 AI
호출 없이 곧바로 로컬에서 처리됩니다(자기학습으로 비용이 점점 줄어듭니다).

**켜기(기본은 꺼짐)**

- 이 기능은 **선택 옵션이며 기본값은 꺼짐**입니다. [5. 설정 화면](#5-설정-화면)의
  **"AI 학습"** 항목을 켜야 동작합니다.
- 학습은 문장을 표준 문형으로 바꾸기 위해 AI를 쓰므로, [6. LLM 해석
  백엔드](#6-llm-해석-백엔드)가 **API 키** 또는 **구독 CLI**로 준비돼 있어야
  합니다(끄기 상태에서는 학습도 동작하지 않습니다). 표준 문형 변환에는
  구독 CLI(Claude 구독 계정)를 권장합니다.

**쓰는 순서**

1. 해석되지 않은 문장의 확인 카드에서 **[AI로 분석해서 배우기]** 버튼을
   누릅니다.
2. AI가 문장을 표준 문형으로 다시 쓴 결과와, 그것을 해석한 규칙 **미리보기**가
   나타납니다(원문 → 표준 문형 → 규칙).
3. 내용을 확인하고 **저장(승인)**하면 (원문 → 규칙) 매핑이 애드온에
   학습됩니다. 이후 같은 문장이나 아주 비슷한 문장은 AI 없이 즉시 처리되며,
   확인 카드에 출처가 **"학습됨"**으로 표시됩니다.

**안전장치**

- **없는 것은 만들지 않습니다.** AI에게는 인벤토리에 있는 방·기기·모드만
  쓰도록 지시하며, 애드온은 변환된 문장이 **원문에 없던 방·기기·모드를 새로
  지어냈는지** 검사해 그런 경우 저장을 막습니다(환각 방어).
- 학습해서 저장하기 전에 규칙이 실제 엔티티를 가리키는지 다시 검증하고,
  최종 저장은 **사용자가 승인**해야만 이뤄집니다.
- 저장된 학습 항목이 참조하는 기기가 나중에 사라지면(엔티티 삭제 등),
  애드온을 다시 시작할 때 그 학습 항목을 자동으로 정리합니다.

**학습된 패턴 관리**

- [5. 설정 화면](#5-설정-화면)의 **"학습된 패턴"** 목록에서 지금까지 배운
  항목(원문·표준 문형·사용 횟수)을 보고, 필요 없는 항목을 **삭제**할 수
  있습니다.
- 학습 항목은 다른 설정과 함께 애드온 데이터 폴더(`/data/learned_patterns.yaml`)에
  저장되어 Home Assistant 백업에 포함됩니다.

## 7. 자체 규칙 엔진과 데이터 저장

루틴 화면에서 만든 규칙은 Home Assistant의 자동화 저장소(`automations.yaml`,
`automation.*` 엔티티)를 전혀 쓰지 않습니다. 대신 애드온이 자체 엔진으로
동작합니다.

- Home Assistant는 **상태 변화를 알려주는 소스**(WebSocket 구독)와 **기기를
  움직이는 통로**(서비스 호출)로만 쓰입니다.
- 규칙·설정·실행 기록·대기 중인 타이머·모드 상태(켜짐/꺼짐)는 애드온의 영속
  데이터 폴더(`/data`)에 저장됩니다. 이 폴더는 Home Assistant의 애드온 백업
  대상에 포함되므로, 평소 하던 대로 백업을 받으면 루틴과 모드 상태도 함께
  백업·복원됩니다.
- Home Assistant 연결이 잠깐 끊겼다가 복구되면 엔진이 상태를 조용히 다시
  동기화하며, 이 과정에서 규칙이 잘못 발화하지 않습니다.
- 애드온이 재시작돼도 "5분 동안 없으면" 같은 지속 조건의 남은 대기 시간이
  복원됩니다.
- 같은 규칙이 액션 실행 중 오류를 내면 그 규칙에만 오류가 기록되고, 다른
  규칙 실행에는 영향이 없습니다. 자세한 내용은 [9. 문제
  해결](#9-문제-해결faq)의 "오류로 꺼짐" 항목을 참고하세요.

## 8. HA 자동화 탭 (기존 카드 편집기)

**HA 자동화** 탭은 Home Assistant 표준 자동화(`automations.yaml` 기반,
`automation.*` 엔티티)를 만들고 관리하는 화면입니다. 루틴과 달리 여기서 만든
자동화는 Home Assistant의 기본 자동화 목록에도 그대로 나타납니다.

### 8.1 목록 화면

등록된 모든 (HA 표준) 자동화를 카드로 보여줍니다. 각 카드에는 다음이 있습니다.

- 이름(alias)과 켜짐/꺼짐 토글 스위치
- 마지막 실행 시각(없으면 "실행 이력 없음")
- **▶ 실행**: 조건을 무시하고 지금 바로 트리거·액션을 실행(수동 테스트용)
- **편집** / **삭제** 버튼
- 상단의 **+ 새 자동화** 버튼 → 위저드로 이동

`configuration.yaml`이나 별도 YAML 파일에서 손으로 관리하는 자동화(이 애드온이나
Home Assistant 기본 편집기로 만들지 않은 자동화)는 **"YAML 관리형"** 배지가
붙고, 편집·삭제 버튼이 비활성화됩니다. 이런 자동화는 원본 YAML 파일을 직접
수정해야 합니다. 자세한 내용은 [8.6 YAML 관리형 자동화 제한](#86-yaml-관리형-자동화-제한)을 참고하세요.

### 8.2 위저드 — 새 자동화 만들기

"언제 실행할까요?" 화면에서 자동화를 시작할 계기를 4개의 큰 카드 중에서 고릅니다.

| 카드 | 계기 | 내부적으로 준비되는 트리거 |
|---|---|---|
| ⏰ 시간이 되면 | 특정 시각/주기/일출·일몰 | 시각(time) 트리거 |
| 🚶 사람이 오가면 | 구역 출입, 재실 여부 | 구역(zone) 트리거 |
| 📡 센서가 감지하면 | 모션·문열림·누수 등 이진 감지기 | 상태(state) 트리거 |
| 📊 값이 변하면 | 온도·습도·전력 등 수치 변화 | 수치(numeric_state) 트리거 |

카드를 고르면 해당 유형의 트리거 폼이 미리 채워진 채로 편집기로 이동합니다.
어떤 카드에도 맞지 않으면 하단 **"고급: 직접 구성"** 을 눌러 빈 편집기에서
원하는 트리거부터 자유롭게 추가할 수 있습니다.

### 8.3 편집기 — 카드 스택

편집기는 세로로 쌓인 3개의 카드로 구성됩니다.

1. **`~할 때`(트리거)**: 자동화를 시작시키는 계기들. 트리거를 2개 이상
   추가하면 "이 중 하나라도 일어나면"이라는 안내 문구가 표시됩니다(OR 결합,
   Home Assistant 트리거의 기본 동작).
2. **`~인 동안`(조건, 선택)**: 트리거가 발생해도 액션을 실행하려면 추가로
   만족해야 하는 조건. **모든 조건 충족(AND) / 하나라도 충족(OR)** 세그먼트
   토글로 결합 방식을 바꿀 수 있습니다. 조건이 없으면 트리거만으로 바로
   실행됩니다.
3. **`실행`(액션)**: 순서대로 실행할 동작 목록. 항목을 위/아래로 이동할 수
   있고, **딜레이 추가** 버튼이 항상 눈에 띄게 노출되어 있습니다.

각 카드의 **+ 추가** 버튼을 누르면 모달이 열리고, **기본** 탭(자주 쓰는
타입 목록)과 **고급** 탭(분기·반복·대기 등)을 오갈 수 있습니다. 고급 탭은
액션 카드에서만 choose/if/repeat/parallel/wait/stop 유형을 제공합니다.
각 항목에는 복제 / 삭제 / 위아래 이동 메뉴가 붙습니다.

화면 하단에는 **[YAML 미리보기]**, **[저장]** 버튼이 고정되어 있습니다.
저장을 누르면 확인 모달이 뜨며 다음을 확인/입력합니다.

- 지금 구성으로 만들어질 내용의 한국어 자연어 요약
- 자동화 이름(alias, 필수)
- 실행 모드(mode, 기본값 `single`) — 각 모드가 무슨 뜻인지 툴팁으로 설명
- 가스밸브·잠금장치 등 안전과 관련된 액션이 포함되어 있으면 별도 경고 문구

여기서 확정해야 실제로 저장(생성/수정)됩니다. YAML 미리보기와 요약은 항상
백엔드가 실제로 저장할 내용을 그대로 계산한 결과이므로, 화면에 보이는 것과
실제로 저장되는 자동화가 다르지 않습니다.

### 8.4 엔티티 피커

트리거·조건·액션 어디서든 엔티티를 고를 때 열리는 3단 화면입니다.

1. **방 그리드**: 거실·안방·작은방·주방·현관·베란다·욕실 등 방 이름과
   엔티티 개수. 방에 배정되지 않은 엔티티는 **"미배정"** 그룹에서 찾습니다.
2. **카테고리 칩**: 고른 방에 실제로 존재하는 카테고리만 보여줍니다
   (조명/스위치·콘센트/안전/감지기/환경 센서/난방·공조/환기·팬/커튼·블라인드/
   미디어/잠금/사람·위치/기타).
3. **엔티티 목록**: 이름, 현재 상태, 소속 방·기기 정보를 함께 보여주는
   리스트에서 최종 선택합니다.

상단 검색창은 전 단계 무시하고 전역 검색(부분 일치 + 초성)을 합니다. 한글
초성만 입력해도(`ㄱㅅ` → "거실") 관련 엔티티를 찾을 수 있습니다. 트리거/조건
목적으로 호출된 피커는 필요한 경우 도메인이나 device_class로 목록을 미리
좁혀서 보여줄 수 있습니다(예: 구역 트리거는 `person.*`만 표시).

이 엔티티 피커는 루틴 화면의 해석 확인 카드에서 "해석 못 함" 칩을 탭했을 때도
그대로 재사용됩니다.

### 8.5 트리거 · 조건 · 액션 타입

#### 트리거

| 유형 | 설명 |
|---|---|
| 상태(state) | 엔티티 상태가 (특정 값에서) 특정 값으로 바뀔 때. `~인 채로 유지` 시간 지정 가능 |
| 수치(numeric_state) | 센서 등 수치 값이 기준치 이상/이하가 될 때 |
| 시각(time) | 매일 지정한 시각(`HH:MM` 또는 `HH:MM:SS`)에 |
| 주기(time_pattern) | 몇 시간/분/초마다 반복(예: 5분마다) |
| 일출·일몰(sun) | 해가 뜨거나 질 때, 앞뒤 오프셋 지정 가능 |
| 구역 출입(zone) | 사람(person) 엔티티가 특정 구역에 들어오거나(enter) 벗어날 때(leave) |
| 템플릿(template) | Jinja2 템플릿 조건식이 참이 될 때(고급) |
| 홈어시스턴트 이벤트(homeassistant) | Home Assistant가 시작(start)되거나 종료(shutdown)될 때 |

#### 조건

| 유형 | 설명 |
|---|---|
| 상태(state) | 엔티티가 지정한 상태값과 같은지 |
| 수치(numeric_state) | 값이 지정한 범위(이상/이하) 안에 있는지 |
| 시간대(time) | 지정한 시각 범위·요일에 해당하는지 |
| 일출·일몰(sun) | 일출/일몰 기준 시간대에 해당하는지 |
| 구역(zone) | 사람 엔티티가 특정 구역 안에 있는지 |
| 템플릿(template) | Jinja2 조건식이 참인지(고급) |
| 트리거 조건(trigger) | 여러 트리거 중 지정한 번호의 트리거로 실행됐는지 |
| AND / OR / NOT | 하위 조건들을 묶어 복합 조건을 만드는 고급 유형 |

#### 액션

| 유형 | 설명 |
|---|---|
| 기기/서비스 호출(service) | 조명 켜기, 온도 설정 등 실제 기기·서비스 호출 |
| 지연(delay) | 시:분:초만큼 다음 동작 전까지 대기 |
| 조건 대기(wait_template) | 템플릿 조건이 참이 될 때까지 대기(시간 제한 가능) |
| 트리거 대기(wait_for_trigger) | 지정한 트리거가 발생할 때까지 대기 |
| 중간 조건(condition) | 이 지점의 조건이 거짓이면 이후 동작 중단 |
| 상황별 분기(choose) | 여러 조건 묶음 중 맞는 것을 골라 서로 다른 동작 실행 |
| 조건부 실행(if-then-else) | 조건이 참일 때/거짓일 때 다른 동작 실행 |
| 반복(repeat) | 횟수 지정 또는 조건(while/until) 기반 반복 실행 |
| 동시 실행(parallel) | 여러 동작 묶음을 동시에 실행 |
| 자동화 중지(stop) | 사유 메시지와 함께 이후 동작을 중단 |

> 참고: 루틴(자연어) 화면의 규칙은 위 유형 중 일부(state, numeric_state,
> service, delay 등)에 더해 "5분 동안 없으면" 같은 지속 조건, "새벽시간에"
> 같은 시간대 조건 등 루틴 전용 표현을 추가로 지원합니다. 반대로 일출·일몰,
> 템플릿(Jinja2) 조건은 루틴 문장으로는 만들 수 없고 HA 자동화 탭에서만
> 사용할 수 있습니다.

### 8.6 YAML 관리형 자동화 제한

Home Assistant의 자동화는 크게 두 갈래로 관리됩니다.

- **UI/ID 기반 자동화**: Home Assistant 기본 자동화 편집기나 이 애드온으로
  만든 자동화. 내부적으로 고유 ID(`attributes.id`)가 있어 API로 조회·수정·
  삭제가 가능합니다. 이 애드온이 편집할 수 있는 대상은 이 자동화들뿐입니다.
- **YAML 관리형 자동화**: `configuration.yaml` 등에서 사람이 직접 작성한
  자동화로, 위 ID가 없는 경우가 많습니다. 목록 화면에서 **"YAML 관리형"**
  배지로 표시되며, 이 애드온에서는 편집·삭제 버튼이 비활성화됩니다. 이런
  자동화를 고치려면 원본 YAML 파일을 직접 수정한 뒤 Home Assistant 설정을
  다시 불러오거나 재시작해야 합니다.

이 애드온에서 새로 만든 자동화는 항상 ID가 부여되므로, 이후에도 계속 이
화면에서 편집할 수 있습니다.

## 9. 문제 해결(FAQ)

### 루틴 화면

**Q. 헤더의 연결 상태 점이 빨간색이에요.**
A. 애드온과 Home Assistant 사이의 상태 구독(WebSocket) 연결이 끊긴
상태입니다. 자동으로 재연결을 계속 시도하므로 잠시 기다렸다가 30초 뒤 자동
갱신되는지 확인하세요. 계속 빨간색이면 Home Assistant 코어가 재시작 중이거나
네트워크 문제가 없는지 확인하고, 필요하면 애드온을 재시작하세요. 연결이
끊긴 동안에도 저장된 규칙은 그대로 남아 있고, 재연결되면 자동으로 상태를
다시 동기화합니다.

**Q. 루틴 카드에 "오류로 꺼짐" 배지가 떴어요. 다시 켜려면 어떻게 하나요?**
A. 같은 규칙이 실행 중 오류를 3회 연속 내면 안전을 위해 자동으로 꺼집니다
(`auto_disabled`). 카드 우측의 활성 토글을 **한 번 꺼졌다가 켜지도록 다시
탭**하세요(이미 켜짐으로 보여도 한 번 껐다 켜면 됩니다) — 그러면 오류 기록과
꺼짐 상태가 초기화되고 규칙이 다시 정상적으로 실행됩니다. 다시 켜기 전에
카드 메뉴의 **▶ 테스트 실행**으로 액션이 지금은 정상적으로 도는지 먼저
확인해 보는 것을 권장합니다. 오류가 반복되면 문장이 가리키는 엔티티가
삭제되었거나 서비스 호출 대상이 바뀌지 않았는지 확인하세요.

**Q. 문장을 해석했는데 일부가 빨간 점선으로 나와요(해석 못 함).**
A. 애드온이 그 구간에 해당하는 엔티티나 값을 찾지 못한 상태입니다. 해당
칩을 탭해 엔티티 피커에서 직접 골라주세요. 자주 등장하는 표현이라면
**설정 → 별칭**에 표면형과 엔티티를 등록해 두면 다음부터 자동으로
해석됩니다.

**Q. 후보가 여러 개 뜨는(노란 테두리) 칩은 왜 그런가요?**
A. 이름이 비슷한 엔티티가 여러 개이거나, 확신도가 충분히 높지 않을 때
후보 목록으로 표시됩니다. 원하는 항목을 고르면 됩니다.

**Q. LLM 해석 백엔드가 "준비됨"으로 안 떠요.**
A. 먼저 애드온 옵션(구성 탭)에서 `llm_backend`를 `api` 또는 `cli`로
선택했는지 확인하세요(기본값은 `off`입니다). `api`를 골랐다면 Anthropic API
키를 입력했는지, `cli`를 골랐다면 `claude setup-token`으로 발급받은 구독
토큰을 붙여넣었는지 확인하세요(설정 → LLM 해석 백엔드, [6. LLM 해석
백엔드](#6-llm-해석-백엔드) 참고). 값이 없으면 그렇게 표시된 것이
정상이며, 백엔드가 준비되어 있어도 로컬 파서가 이미 충분히 확신하는
문장에는 호출되지 않는 것이 정상입니다(불필요한 호출을 줄이기 위함).

**Q. 애드온 옵션에서 구독 CLI를 선택했는데 적용이 안 되는 것 같아요.**
A. 최종적으로 적용되는 백엔드는 애드온 옵션이 아니라 **설정 화면**(루틴 →
설정 → LLM 해석 백엔드)에서 라디오로 고르고 [설정 저장]을 누른 값입니다.
설정 화면에서 아직 한 번도 저장한 적이 없을 때만 애드온 옵션의 값이 대신
쓰입니다. 애드온 옵션에서 구독 CLI나 API 키를 선택했다면, 설정 화면에서도
같은 항목을 라디오로 고른 뒤 저장해야 계속 유지됩니다.

**Q. 모드를 껐다 켰다 해도 규칙이 반응하지 않아요.**
A. 설정 → 모드 관리에서 토글로 상태를 바꿨는지, 그리고 규칙 문장이 그 모드
이름을 정확히 트리거로 쓰고 있는지(예: "슬립모드가 켜지면")를 확인하세요.
모드 이름을 조건으로만 쓴 규칙("…슬립모드이고…")은 모드가 바뀌는 순간이
아니라 다른 트리거가 함께 만족될 때만 발화합니다.

**Q. "직접 지정"으로 만든 규칙이 저장되지 않아요.**
A. 하단 요약 위에 "아직 저장할 수 없어요" 안내가 있다면, 각 규칙(서브룰)에
트리거 역할을 가진 단어와 액션(동작) 역할을 가진 단어가 최소 하나씩 있는지
확인하세요. 여러 규칙으로 나누고 싶다면 경계가 되는 단어에 "여기서 규칙
나눔"을 켰는지도 확인하세요.

**Q. 루틴이 저장되지 않고 [루틴 저장] 버튼이 비활성화돼요.**
A. 해석 확인 카드에 "해석 못 함"(빨간 점선) 상태인 칩이 남아 있으면 저장할
수 없습니다. 모든 칩을 확정(확정 또는 애매함이라도 후보 선택)한 뒤 다시
시도하세요.

### HA 자동화 탭

**Q. 자동화를 저장했는데 목록에 보이지 않아요.**
A. 저장은 Home Assistant Core의 자동화 설정 API를 통해 이루어지며 저장과
동시에 자동으로 다시 불러와집니다. 그래도 반영되지 않는다면
`configuration.yaml`에 자동화 관리 설정이 아래처럼 올바르게 되어 있는지
확인하세요.

```yaml
automation: !include automations.yaml
```

이 항목이 없거나, 자동화를 다른 파일에서 완전히 분리해 커스텀 방식으로
로드하고 있다면 API가 기록하는 위치와 실제로 읽어들이는 파일이 어긋날 수
있습니다. 설정을 확인한 뒤에는 **개발자 도구 → YAML → 자동화 다시 불러오기**
를 실행하거나 Home Assistant를 재시작하세요.

**Q. 가스밸브/잠금장치 관련 자동화를 저장하려니 경고가 떠요.**
A. 가스, 잠금장치 등 안전과 직결되는 동작이 포함된 자동화는 저장 확인
모달에 경고 문구가 추가로 표시됩니다. 의도한 동작이 맞는지 다시 확인한 뒤
저장하세요.

**Q. 자동화가 켜져 있는데 실행되지 않아요.**
A. 목록 화면에서 **▶ 실행** 버튼으로 트리거 없이 바로 실행되는지 먼저
확인하세요. 조건 카드에 조건이 걸려 있다면, 트리거가 발생한 시점에 그
조건이 충족되지 않아 액션까지 진행되지 않았을 수 있습니다.

### 공통

**Q. 사이드바에 "자동화 메이커" 패널이 안 보여요.**
A. 애드온이 시작(Start) 상태인지, 애드온 정보 화면의 "사이드바에 표시"
옵션이 켜져 있는지 확인하고 브라우저를 새로고침하세요.

**Q. "허용되지 않은 접근입니다"(403) 오류가 나요.**
A. 이 애드온의 백엔드는 Home Assistant Ingress 게이트웨이를 통한 접근만
허용합니다. 애드온의 "웹 UI 열기" 버튼이나 사이드바 패널로만 접속하세요.
또한 Ingress 패널은 관리자 계정 전용이므로, 비관리자 계정으로 로그인했다면
관리자 계정으로 다시 로그인해 보세요.

**Q. 찾는 엔티티가 목록(엔티티 피커)에 없어요.**
A. 다음 엔티티는 자동화를 만들 때 의도적으로 목록에서 제외됩니다.

- 사용 중지(비활성화)된 엔티티
- "진단" 또는 "설정" 카테고리로 분류된 엔티티
- 통합구성요소가 숨김 처리한 엔티티
- `automation`, `update`, `tts`, `stt`, `conversation`, `assist_satellite`,
  `zone`, `persistent_notification` 도메인 엔티티

방(영역)에 아직 배정되지 않은 엔티티는 방 그리드의 **"미배정"** 그룹에서
찾을 수 있습니다.

**Q. 엔티티 이름이 이상하게(엔티티 ID 그대로) 표시돼요.**
A. Home Assistant에서 해당 엔티티에 별칭(친숙한 이름)이 지정되지 않은
경우입니다. **설정 → 기기 및 서비스 → 엔티티**에서 이름을 지정해 주세요.
루틴 화면에서는 대신 **설정 → 별칭**에 원하는 표면형을 따로 등록할 수도
있습니다.
