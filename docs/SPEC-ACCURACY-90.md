# SPEC-ACCURACY-90 — 오프라인(L1+L2) 정직 정확도 90% 로드맵

> 상태: 설계 확정 문서(리서치 총괄 산출). 근거는 6갈래 플랫폼 리서치(SmartThings · Fibaro HC3 ·
> Homey · Apple HomeKit · Google/Alexa/Bixby 음성비서 · 한국어 표면형 변이)와 코드 실측
> (`backend/nl/parser.py`·`pattern_match.py`, `tools/corpus/evaluate_hybrid.py`,
> `out/hybrid_report.md`)이다. 수치는 전부 실측 또는 "추정"으로 명시한다.

## 0. 목표와 절대 제약

- **목표: held-out(확장판) test split에서 오프라인(L1 결정적 규칙 + L2 템플릿 라이브러리)만으로
  exact ≥ 90%.**
- **CLI(L3)는 롱테일 보조**: 실사용 발동률 <10%, 정확도 기여 <5%p를 합격 기준에 포함한다.
  90%가 CLI에 기대면 실패로 판정한다.
- 정직성 원칙(SPEC-ACCURACY §0) 유지: held-out은 라이브러리 구축에 절대 쓰지 않고,
  τ 등 하이퍼파라미터는 dev에서만 튜닝, test 동결.

## 1. 현재 상태 (실측, held-out 205문장)

| 단계 | test | 전체 |
|---|---|---|
| L1 (앱 파서) | 16.8% | 14.6% |
| L1A (+A/B 결정적 규칙 오버레이) | **69.0%** | 64.9% |
| +L2 (템플릿 매처) | 69.0% (**기여 0**) | 64.9% |
| +L3 (CLI) | 75.2% | 73.2% |

- solved_by: rule 30 · ruleA 103 · **template 0** · cli 17 · none 55. 이득의 사실상 전부가
  결정적 규칙이다.
- 영역별 오프라인(L1A) 최하: misparse 7.1% · multiclause 29.2% · time 50% · para_hard 53.7%.
- L2 매처: needs_help 중 후보 산출 5건(L1A-ok 보호로 미시도 8건), exact 0건 → 구조상한 0%.

**격차 산정**: 90% 오프라인 = test에서 +21%p. 신규 확장 held-out(520문장, §4)에서는 기준선이
더 낮게 나올 것이므로(신규 의미축 포함), 실질 격차는 25~30%p로 본다. 주력은 L1 확장(추정
+15~20%p), L2 재설계(추정 +3~6%p), 잔여는 핵심 잔차 정리다.

## 2. 측정 방법론

1. **정직 지표 = 확장 held-out test split의 오프라인(L1A+L2) exact%.** +L3은 참고 지표로 병기.
2. dev/test는 기존 md5(sentence)%2 분할 유지. 신규 카테고리도 동일 규칙.
3. held-out 문장·gold는 라이브러리(templates/slots/paraphrases train/pattern_library/
   learned_patterns 프리워밍)에 노출 금지. 리서치 new_case_candidates를 시드로 쓰되
   **문장을 새로 작성**한다(시드 문장 자체는 라이브러리 편입 가능 — 겹치는 문장은 held-out에서 제외).
4. `evaluate_hybrid.py` 상설 지표 추가: L2 shadow 지표(시도/채택/exact/회귀), delexicalize
   미인식 어절률, CLI 발동률·기여. 회귀(regression) 0 계약은 전 단계 유지.
5. 금지문 적대 세트는 exact가 아니라 **"위험 오동작 0"**(정반대 액션 미생성)으로 별도 판정.

설계 공통 원칙:
- **자체 룰 엔진**이므로 노드 신설(sun/weekday/presence_agg/time_pattern/repeat)은 엔진+빌더+
  파서+테스트를 한 단위로 커밋한다.
- 한국어 중의는 파서가 단정하지 않고 **확인 칩**으로 노출한다: '넘으면'(엣지/레벨),
  '따뜻하게'(난방/색온도), '알려줘'(폰/스피커), '7시쯤', '껐다 켜'(재시작/토글).
- 금지문(-지 마/못 -게)은 정반대 액션 생성이 최악이므로, 커버 이전에도 **오독 방어가 선행**한다.

## 4. held-out 확장 (205 → 520문장)

- 기존 151(heldout) + 54(para_hard) 유지. 신규 ~315문장을 16개 카테고리로 추가(카테고리당 15~25):
  sun-offset · weekday-calendar · interval-repeat · negation-exception · prohibition(적대) ·
  aspect-condition(대비 쌍) · presence-quantifier · numeric-edge-between · quoted-message ·
  action-params · duration-revert · event-trigger · surface-variation(기존 gold의 수사·어미·존칭 변형) ·
  postposed-clause · weather-external · else-branch.
- 작성 규칙: 리서치 ko_phrasings/new_case_candidates를 시드로 하되 문장은 재작성, gold는
  mock_data 실존 엔티티로 손라벨 + `validate_rule_model` 자체검증. difficulty 3단 유지.
- weather-external 등 목표 밖 카테고리도 포함해 **의도적으로 미달을 측정**한다 — 90%가
  '쉬운 것만 골라 채운 90%'가 되지 않도록.

## 5. L2 매처 재설계 — 기여 0 진단과 처방

**진단(실측)**: ① 발동 창 협소 — "L1A ok면 억제" 정책으로 매처는 not-ok 파탄 문장(대부분
라이브러리에 gold가 없는 미지 의미축)만 봄 → 볼 수 있는 문장과 풀 수 있는 문장이 서로소.
② 라이브러리가 파서의 그림자 — covered 골격 전부가 L1이 이미 푸는 templates.yaml 유래라
'아는 의미의 새 어순' 골격이 없음. ③ delexicalize 빈약 — `_sym()` 심볼 사전이 작고 극성
멀티셋 **완전일치** 게이트가 구어 변이(느껴지면/인기척/켜지게 해줘)에서 후보를 τ 판정 이전에
전멸시킴. 부가로 절 단위 매칭 부재(multiclause에서 LCS 급락), 입력/템플릿 스트림 생성 경로
이원화.

**처방 5축**:
1. **게이트 재설계(shadow-try)**: L1A ok + conf<0.6이어도 매처를 시도하되, 후보가 L1A 모델과
   구조 동형이면 무시(합의), 다르면 검증 게이트(엔티티 실존·도메인·서브룰 수) 통과 시에만 채택.
   shadow 지표를 리포트에 상설, 회귀 0 계약 유지.
2. **라이브러리 독립화**: paraphrases train 골격 자동 편입(delexicalize→골격+gold, 수백 개) +
   신규 의미축 템플릿을 L1 확장과 동시 등재 + `/data/learned_patterns.yaml` 런타임 통합.
3. **delexicalize v2**: 동사 어간→심볼 분류표 확장, 사역 wrapper 정규화(§3-1 전처리 공유),
   미인식 어절률 지표화, 입력/템플릿 스트림 생성 단일화.
4. **절 단위 매칭**: `parser._find_pivots` 재사용, 서브룰별 대조 후 조립.
5. **게이트 완화 + 재캘리브레이션**: 극성 멀티셋 완전일치 → 불일치 심볼당 가중 패널티,
   τ는 확장 라이브러리 위 dev 재스윕(test 동결). 마진·도메인·부분집합 게이트 유지.

L2의 역할 정의: **L1이 아는 의미의 새 어순 흡수**(주 수혜: multiclause·para_hard) +
**learned 패턴 실행기**(CLI 증류 수용체). 목표 순리프트 +3~6%p, 회귀 0.

## 6. CLI(L3) 최소화

1. 발동: L1 needs_help && L2 miss && 사용자 명시 선택("AI로 분석해서 배우기")만. 자동 발동 없음.
2. 증류: 성공 시 (원문→정규형→gold)를 승인 후 learned_patterns.yaml append → 동일 유형은
   이후 L2 로컬 처리. 롱테일 유형당 호출 상한 1회의 복리 구조.
3. 프리워밍: 배포 전 gap_library+리서치 롱테일을 CLI 배치 정규화해 정적 라이브러리 동봉.
4. 캐시 우선(cli_cache.jsonl) + 예산제(--cli-budget) 유지.
5. 계약: 오프라인 test ≥90% · CLI 발동률 <10% · CLI 기여 <5%p — hybrid_report 상설 지표.

## 8. 리스크

- **비중 추정 오차**: §3의 est share는 추정 — Phase 0 라벨링 후 실측으로 교체하고 우선순위 재조정.
- **금지문 오독**: 커버 확대 중 최악의 안전 리스크. Phase 2를 신규 의미축보다 앞세우고 적대
  세트를 상설 회귀로 고정.
- **L2 게이트 완화의 회귀 위험**: shadow-try는 채택 검증 게이트와 회귀 0 계약으로 이중 방어.
  완화 후 l2_regressions>0이면 즉시 롤백.
- **held-out 오염**: 라이브러리 자동 편입(paraphrases train·프리워밍)이 커질수록 오염 위험 —
  편입 파이프라인에 held-out 문장 해시 블랙리스트 검사 추가.
- **엔진 노드 신설 부담**: sun/weekday 등은 파서만이 아니라 엔진·빌더·프론트 칩까지 관통 —
  축별 수직 슬라이스로 커밋해 부분 완성 상태를 남기지 않는다.
- **확인 칩 남발**: 중의 해소를 칩에 과도하게 미루면 UX 저하 — 칩은 화자 편차가 실재하는
  축(시각 근사·따뜻하게·알려줘)에 한정.
- **90% 이후의 과적합 착시**: 확장 held-out도 결국 저작물 — 실사용 로그 기반 재측정(learn API
  수집 문장)을 차기 사이클의 held-out 후보로 예약한다.
